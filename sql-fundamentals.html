Yogananda Grocery Shop & The World of Databases

Meet Yogananda, a grocery shop owner in Chennai. He sells rice, sugar, oil, and biscuits – everything. He has a big problem with maintaining all the data on paper.

  • One notebook for customer details
  • One notebook for sale.
  • One notebook for payment details (EMI's type).

When a customer asks, "Did I pay last week?". If 5 customers came together, everything became a mess! Then one day, his son said "Appa, let's use a computer database!"

What is Database?

A database is like a digital notebook—a structured place to store, manage, and update information easily. It's organized neatly using tables with rows and columns.

Example: One table for customers, one table for products.

Benefits for Yogananda with Database:

Problem (Before) Solution (With Database)
Flipping pages Data is neatly stored in tables
Can't find old sales Instantly search and find details
Price update takes time Update once – reflected everywhere.
Data not safe Password protected and secure.

Now, Yogananda can talk to his customer and ask questions like:

  • Show me all customers from Chennai

But... how will he talk to the database?

Just like we speak English to people, computers understand a special language called SQL.

You can practice writing SQL queries using this online SQL compiler: SQL Compiler ↗

Structured Query Language (SQL):

SQL is a language used to communicate with databases. It helps you:

Action SQL Command Example
Create tables CREATE Create a "customers" table
Add data INSERT Add new customer
Change data UPDATE Update customer's city
Remove data DELETE Delete old records
Retrieve data SELECT Show list of customers

SQL Commands:

To manage all this, SQL gives us five families of commands, called SQL Sub-Languages.

1. DDL (Data Definition Language):

These commands are used to design or modify the structure of databases and tables — they deal with how data is stored, not the actual data itself.

Command Use Syntax
CREATE Make a new table or database CREATE TABLE table_name (column_name datatype, ...);
ALTER Add, modify, or delete columns in an existing table ALTER TABLE table_name ADD column_name datatype;
DROP Permanently delete a table or database DROP TABLE table_name;
TRUNCATE Remove all data from a table but keep the structure TRUNCATE TABLE table_name;
RENAME Rename an existing table RENAME TABLE old_name TO new_name;

Now, let's apply these DDL commands on our sample database structure. Note that DDL only defines the structure of tables — it does not insert or modify any data. Adding records (data) is done later using DML commands such as INSERT. 

    
-- Create a new database
CREATE DATABASE GroceryShop;

-- Create the Customers table (only structure, no data yet)
CREATE TABLE Customers (
    CustomerID INT PRIMARY KEY,
    CustomerName VARCHAR(50),
    City VARCHAR(50)
);

-- Create the Orders table linked to Customers (structure only)
CREATE TABLE Orders (
    OrderID INT PRIMARY KEY,
    CustomerID INT,
    Product VARCHAR(50),
    FOREIGN KEY (CustomerID) REFERENCES Customers(CustomerID)
);

-- Note:
-- The above statements only define table structure.
-- To insert data into these tables, we use DML commands such as:
-- INSERT INTO Customers VALUES (1, 'Yogananda', 'Chennai');

ALTER

Used to add or modify columns in an existing table.

    
-- Add a new column for storing phone numbers
ALTER TABLE Customers ADD Phone VARCHAR(10);

-- Modify column size
ALTER TABLE Customers MODIFY CustomerName VARCHAR(100);

RENAME

Used to change the table name.

    
-- Rename Customers table to Clients
RENAME TABLE Customers TO Clients;

TRUNCATE

Used to delete all rows from a table while keeping its structure intact.

    
-- Remove all order records but retain table structure
TRUNCATE TABLE Orders;

DROP

Used to permanently delete a table or database.

    
-- Drop the Orders table permanently
DROP TABLE Orders;

-- Drop the entire database
DROP DATABASE GroceryShop;

2. DML (Data Manipulation Language):

DML commands are used to manage and manipulate the actual data stored within database tables.

Think of DML as "filling, updating, or clearing the shelves" — it deals with the data itself, not the structure.

Command Use Syntax
INSERT Add new rows of data into a table INSERT INTO table_name (column1, column2) VALUES (value1, value2);
UPDATE Modify existing data within a table UPDATE table_name SET column1 = value1 WHERE condition;
DELETE Remove data from a table DELETE FROM table_name WHERE condition;

Let’s now understand each DML command with clear syntax and important notes for practical usage.

INSERT

Used to add new records into a table.


-- Insert multiple customer records
INSERT INTO Customers (CustomerID, CustomerName, City)
VALUES 
(1, 'Yogananda', 'Chennai'),
(2, 'Vishnu', 'Hyderabad'),
(3, 'Ramesh', 'Bangalore'),
(4, 'Kavitha', 'Pune'),
(5, 'Suresh', 'Delhi'),
(6, 'Meena', 'Kochi');

-- Insert corresponding order records
INSERT INTO Orders (OrderID, CustomerID, Product)
VALUES 
(101, 1, 'Sugar'),
(102, 2, 'Rice'),
(103, 3, 'Milk'),
(104, 4, 'Wheat'),
(105, 5, 'Tea'),
(106, 6, 'Coffee');

Important: The number of columns and values must match, and data types should be compatible with the table definition.

UPDATE

Used to modify existing records in a table based on a condition.


-- Update city for a specific customer
UPDATE Customers
SET City = 'Hyderabad'
WHERE CustomerID = 3;

-- Update product name for a specific order
UPDATE Orders
SET Product = 'Brown Sugar'
WHERE OrderID = 101;

Important: Always use a WHERE clause; omitting it updates all rows in the table.

DELETE

Used to remove one or more records from a table.


-- Delete a customer record
DELETE FROM Customers
WHERE CustomerID = 5;

-- Delete all orders for a specific customer
DELETE FROM Orders
WHERE CustomerID = 5;

Important: If you skip the WHERE clause, all data from the table will be deleted.

3. DQL (Data Query Language):

DQL commands are used to query and retrieve data stored in database tables.

Think of DQL as "asking your database questions and getting answers" — it helps you view and analyze the stored information.

Command Use Syntax
SELECT Retrieve specific or all records from one or more tables SELECT column1, column2 FROM table_name WHERE condition;

Let’s use the previously inserted Customers and Orders tables to perform various DQL queries and understand how to retrieve data effectively.

1. Retrieve All Records

Fetch all columns and rows from the Customers table.


-- Retrieve all customers
SELECT * FROM Customers;

-- Retrieve all orders
SELECT * FROM Orders;

2. Retrieve Specific Columns

Fetch only the required columns instead of the entire table.


-- Retrieve only customer names and cities
SELECT CustomerName, City FROM Customers;

-- Retrieve only product names from orders
SELECT Product FROM Orders;

3. Retrieve Data with Conditions

Use the WHERE clause to filter data based on specific conditions.


-- Customers from Hyderabad
SELECT * FROM Customers WHERE City = 'Hyderabad';

-- Orders made by CustomerID = 3
SELECT * FROM Orders WHERE CustomerID = 3;

4. Retrieve Data in Sorted Order

Use the ORDER BY clause to display results in ascending or descending order.


-- Customers sorted by name
SELECT * FROM Customers ORDER BY CustomerName ASC;

-- Orders sorted by product name (descending)
SELECT * FROM Orders ORDER BY Product DESC;

5. Retrieve Data Using JOIN

Use JOIN to combine related data from multiple tables based on a common column.


-- Retrieve customer names with their corresponding products
SELECT 
    C.CustomerName, 
    C.City, 
    O.Product
FROM Customers AS C
INNER JOIN Orders AS O
ON C.CustomerID = O.CustomerID;

Important: The SELECT statement does not modify data — it only retrieves it. Always use filters or joins carefully to avoid large, unnecessary data loads.

4. DCL (Data Control Language):

These commands are used to control who can do what.

Think of DCL as "deciding who can enter which part of your shop".

Command Use Example
GRANT Give permission GRANT SELECT, INSERT ON Customers TO Devendra;
REVOKE Take back permission REVOKE INSERT ON Customers FROM Devendra;

5. TCL (Transaction Control Language):

These commands are used for handling multiple operations safely.

Think of TCL as "undo/redo for your database."

Command Use Example
COMMIT Save changes permanently COMMIT;
ROLLBACK Undo changes (if not committed) ROLLBACK;
SAVEPOINT Create a checkpoint SAVEPOINT sp1;
SET TRANSACTION Set read/write mode SET TRANSACTION READ ONLY;

JOINS – Combining Data from Multiple Tables

Yogananda now has two tables

Customers Table

CustomerID CustomerName City
1 Yogananda Chennai
2 Vishnu Hyderabad
3 Kiran Bangalore

Orders Table

OrderID CustomerID Product Quantity
101 1 Sugar 2
102 2 Rice 1
103 1 Oil 1
104 4 Salt 5

Notice that CustomerID=4 (in Orders Table) doesn't exist in the customers table - maybe an order by mistake. Now Yogananda wants to view "which customers bought what?" To do this, we need JOINS.

What are JOINS?

A JOIN combines rows from two or more tables based on a related column between them (usually Primary Key = Foreign Key).

It's like saying "Hey database, show me information from both notebooks together, where the entries match!"

Types of JOINS:

There are 4 main types you'll use most often.

1. INNER JOIN:

Shows only the matching rows from both tables. If a customer doesn't have an order, or an order doesn't have a valid customer – those rows are ignored.

SELECT Customers.CustomerName, Orders.Product, Orders.Quantity
FROM Customers
INNER JOIN Orders
ON Customers.CustomerID = Orders.CustomerID;

Note: Use INNER JOIN when you only want rows that have matches on both tables.

2. LEFT JOIN:

Returns all rows from the left table (Customers) and the matching rows from the right table (Orders). If there's no match, NULL is shown.

SELECT Customers.CustomerName, Orders.Product, Orders.Quantity
FROM Customers
LEFT JOIN Orders
ON Customers.CustomerID = Orders.CustomerID;

Kiran didn't place the order -> Product and Quantity show NULL.

Note: Use LEFT JOIN when you want all customers, even those without any orders.

3. RIGHT JOIN (or RIGHT OUTER JOIN):

It returns all rows from the right table (Orders) and the matching rows from the left table (Customers).

SELECT Customers.CustomerName, Orders.Product
FROM Customers
RIGHT JOIN Orders
ON Customers.CustomerID = Orders.CustomerID;

Here CustomerName is NULL because OrderID 104 has no valid customer.

Note: Use RIGHT JOIN when you want all orders, even if some customers don't exist.

4. FULL OUTER JOIN:

It returns all rows from both tables, showing NULL where there is no match.

SELECT Customers.CustomerName, Orders.Product
FROM Customers
FULL OUTER JOIN Orders
ON Customers.CustomerID = Orders.CustomerID;

Use FULL JOIN when you need to see everything – matches and non-matches.

SUMMARY:

Join Type What It Shows NULLs Shown?
INNER JOIN Only matching rows No
LEFT JOIN All from left + matches from right Right side NULLs
RIGHT JOIN All from right + matches from left Left side NULLs
FULL JOIN All rows from both tables Both sides NULLs

AGGREGATE FUNCTIONS:

After combining data, Yogananda wants summaries like:

  • How many total customers?
  • How many total orders?
  • Average quantity orders?

That's where Aggregate functions help.

Function Description Example

GROUP BY, HAVING & ORDER BY

Let's go step-by-step.

1. GROUP BY:

Used to group rows having the same value in one or more columns. Usually used with aggregate functions.

Yogananda wants to know: "How many total orders has each customer placed?"

SELECT CustomerID, COUNT(OrderID) AS TotalOrders
FROM Orders
GROUP BY CustomerID;

2. HAVING:

It works like "WHERE", but for groups. It is used to filter results after GROUP BY.

Yogananda wants only those customers who placed more than one order.

SELECT CustomerID, COUNT(OrderID) AS TotalOrders
FROM Orders
GROUP BY CustomerID
HAVING COUNT(OrderID) > 1;

Note: WHERE filters rows before grouping, HAVING filters results after grouping.

3. ORDER BY:

Used to sort results in ascending (ASC) or descending (DESC) order.

SELECT CustomerName, City
FROM Customers
ORDER BY CustomerName DESC;

Note: Default order is ASC.

NORMALIZATION – Making Databases Efficient

Now Yogananda has stored all his shop data in one big table. Everything works fine... until problems appear.

Problem in This Design:

  1. Repetition – Same customer info repeated.
  2. Update anomaly – If Kiran changes his phone, must update every row.
  3. Delete anomaly – If Vishnu's only order is deleted, his info is lost.

To fix this, we apply Normalization.

What is Normalization?

Normalization is the process of organizing data into smaller, related tables to:

  • Remove repetition
  • Reduce storage waste
  • Maintain consistency
  • Improve data integrity

Normal Forms (Levels of Normalization)

a. 1NF (First Normal Form):

Each column must have atomic (single) values. No repeating groups or arrays.

b. 2NF (Second Normal Form)

  • Must be in 1NF.
  • No partial dependency – every non-key column depends on the entire primary key.

c. 3NF (Third Normal Form):

  • Must be in 2NF.
  • No transitive dependency – non-key column shouldn't depend on another non-key column.

SUMMARY OF NORMALIZATION:

Normal Form Rule Fixes
1NF No repeating groups Atomic data
2NF No partial dependency Split based on full key
3NF No transitive dependency Remove derived dependencies
