Yogananda Grocery Shop & The World of Databases

Meet Yogananda, a grocery shop owner in Chennai. He sells rice, sugar, oil, and biscuits – everything. He has a big problem with maintaining all the data on paper.

  • One notebook for customer details
  • One notebook for sale.
  • One notebook for payment details (EMI's type).

When a customer asks, "Did I pay last week?". If 5 customers came together, everything became a mess! Then one day, his son said "Appa, let's use a computer database!"

What is Database?

A database is like a digital notebook—a structured place to store, manage, and update information easily. It's organized neatly using tables with rows and columns.

Example: One table for customers, one table for products.

Benefits for Yogananda with Database:

Problem (Before) Solution (With Database)
Flipping pages Data is neatly stored in tables
Can't find old sales Instantly search and find details
Price update takes time Update once – reflected everywhere.
Data not safe Password protected and secure.

Now, Yogananda can talk to his customer and ask questions like:

  • Show me all customers from Chennai

But... how will he talk to the database?

Just like we speak English to people, computers understand a special language called SQL.

Structured Query Language (SQL):

SQL is a language used to communicate with databases. It helps you:

Action SQL Command Example
Create tables CREATE Create a "customers" table
Add data INSERT Add new customer
Change data UPDATE Update customer's city
Remove data DELETE Delete old records
Retrieve data SELECT Show list of customers

Understanding Tables

Let's connect this to Yogananda's old notebooks.

The CustomerID connects both tables – showing who bought what.

SQL Commands:

To manage all this, SQL gives us five families of commands, called SQL Sub-Languages.

1. DDL (Data Definition Language):

These are used to design or modify tables and databases.

Command Use Example
CREATE Make a new table/database CREATE DATABASE GroceryShop;
ALTER Add or change columns ALTER TABLE Customers ADD Phone VARCHAR(10);
DROP Delete table/database permanently DROP TABLE Customers;
TRUNCATE Delete all data, keep structure TRUNCATE TABLE Customers;
RENAME Rename table RENAME TABLE Customers TO Clients;

2. DML (Data Manipulation Language):

These commands are used to manage data.

Think of DML as "filling or updating the shelves".

Command Use Example
INSERT Add new row INSERT INTO Customers VALUES (1, 'Yogananda', 'Chennai');
UPDATE Change data UPDATE Customers SET City='Hyderabad' WHERE CustomerID=1;
DELETE Remove data DELETE FROM Customers WHERE CustomerID=4;

3. DQL (Data Query Language):

These commands are used to fetch data.

Think of DQL as "asking your system questions and getting answers".

Command Use Example
SELECT Retrieve data SELECT * FROM Customers;

4. DCL (Data Control Language):

These commands are used to control who can do what.

Think of DCL as "deciding who can enter which part of your shop".

Command Use Example
GRANT Give permission GRANT SELECT, INSERT ON Customers TO Devendra;
REVOKE Take back permission REVOKE INSERT ON Customers FROM Devendra;

5. TCL (Transaction Control Language):

These commands are used for handling multiple operations safely.

Think of TCL as "undo/redo for your database."

Command Use Example
COMMIT Save changes permanently COMMIT;
ROLLBACK Undo changes (if not committed) ROLLBACK;
SAVEPOINT Create a checkpoint SAVEPOINT sp1;
SET TRANSACTION Set read/write mode SET TRANSACTION READ ONLY;

JOINS – Combining Data from Multiple Tables

Yogananda now has two tables

Customers Table

CustomerID CustomerName City
1 Yogananda Chennai
2 Vishnu Hyderabad
3 Kiran Bangalore

Orders Table

OrderID CustomerID Product Quantity
101 1 Sugar 2
102 2 Rice 1
103 1 Oil 1
104 4 Salt 5

Notice that CustomerID=4 (in Orders Table) doesn't exist in the customers table - maybe an order by mistake. Now Yogananda wants to view "which customers bought what?" To do this, we need JOINS.

What are JOINS?

A JOIN combines rows from two or more tables based on a related column between them (usually Primary Key = Foreign Key).

It's like saying "Hey database, show me information from both notebooks together, where the entries match!"

Types of JOINS:

There are 4 main types you'll use most often.

1. INNER JOIN:

Shows only the matching rows from both tables. If a customer doesn't have an order, or an order doesn't have a valid customer – those rows are ignored.

SELECT Customers.CustomerName, Orders.Product, Orders.Quantity
FROM Customers
INNER JOIN Orders
ON Customers.CustomerID = Orders.CustomerID;

Note: Use INNER JOIN when you only want rows that have matches on both tables.

2. LEFT JOIN:

Returns all rows from the left table (Customers) and the matching rows from the right table (Orders). If there's no match, NULL is shown.

SELECT Customers.CustomerName, Orders.Product, Orders.Quantity
FROM Customers
LEFT JOIN Orders
ON Customers.CustomerID = Orders.CustomerID;

Kiran didn't place the order -> Product and Quantity show NULL.

Note: Use LEFT JOIN when you want all customers, even those without any orders.

3. RIGHT JOIN (or RIGHT OUTER JOIN):

It returns all rows from the right table (Orders) and the matching rows from the left table (Customers).

SELECT Customers.CustomerName, Orders.Product
FROM Customers
RIGHT JOIN Orders
ON Customers.CustomerID = Orders.CustomerID;

Here CustomerName is NULL because OrderID 104 has no valid customer.

Note: Use RIGHT JOIN when you want all orders, even if some customers don't exist.

4. FULL OUTER JOIN:

It returns all rows from both tables, showing NULL where there is no match.

SELECT Customers.CustomerName, Orders.Product
FROM Customers
FULL OUTER JOIN Orders
ON Customers.CustomerID = Orders.CustomerID;

Use FULL JOIN when you need to see everything – matches and non-matches.

SUMMARY:

Join Type What It Shows NULLs Shown?
INNER JOIN Only matching rows No
LEFT JOIN All from left + matches from right Right side NULLs
RIGHT JOIN All from right + matches from left Left side NULLs
FULL JOIN All rows from both tables Both sides NULLs

GROUP BY, HAVING & ORDER BY

Let's go step-by-step.

1. GROUP BY:

Used to group rows having the same value in one or more columns. Usually used with aggregate functions.

Yogananda wants to know: "How many total orders has each customer placed?"

SELECT CustomerID, COUNT(OrderID) AS TotalOrders
FROM Orders
GROUP BY CustomerID;

2. HAVING:

It works like "WHERE", but for groups. It is used to filter results after GROUP BY.

Yogananda wants only those customers who placed more than one order.

SELECT CustomerID, COUNT(OrderID) AS TotalOrders
FROM Orders
GROUP BY CustomerID
HAVING COUNT(OrderID) > 1;

Note: WHERE filters rows before grouping, HAVING filters results after grouping.

3. ORDER BY:

Used to sort results in ascending (ASC) or descending (DESC) order.

SELECT CustomerName, City
FROM Customers
ORDER BY CustomerName DESC;

Note: Default order is ASC.

NORMALIZATION – Making Databases Efficient

Now Yogananda has stored all his shop data in one big table. Everything works fine... until problems appear.

Problem in This Design:

  1. Repetition – Same customer info repeated.
  2. Update anomaly – If Kiran changes his phone, must update every row.
  3. Delete anomaly – If Vishnu's only order is deleted, his info is lost.

To fix this, we apply Normalization.

What is Normalization?

Normalization is the process of organizing data into smaller, related tables to:

  • Remove repetition
  • Reduce storage waste
  • Maintain consistency
  • Improve data integrity

Normal Forms (Levels of Normalization)

a. 1NF (First Normal Form):

Each column must have atomic (single) values. No repeating groups or arrays.

b. 2NF (Second Normal Form)

  • Must be in 1NF.
  • No partial dependency – every non-key column depends on the entire primary key.

c. 3NF (Third Normal Form):

  • Must be in 2NF.
  • No transitive dependency – non-key column shouldn't depend on another non-key column.

SUMMARY OF NORMALIZATION:

Normal Form Rule Fixes
1NF No repeating groups Atomic data
2NF No partial dependency Split based on full key
3NF No transitive dependency Remove derived dependencies
